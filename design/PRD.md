## 1. 产品背景

### 1.1 行业趋势

近年来，求职市场发生了深刻变革：

- **AI渗透招聘全链路**：超过79%的大型企业使用ATS（Applicant Tracking System）系统自动筛选简历，大量简历在到达HR之前已被机器过滤
- **求职竞争白热化**：平均每个岗位收到200+份简历，求职者必须在极短时间内脱颖而出
- **JD信息不对称**：求职者与招聘方之间存在严重的信息壁垒，用户不清楚自己的简历与岗位的真实差距
- **AI工具普及**：ChatGPT等AI工具的普及让用户开始尝试用AI辅助简历，但缺乏系统化、专业化的一体工具

### 1.2 核心问题

| 问题 | 数据支撑 | 痛点强度 |
|------|---------|---------|
| 简历被ATS系统过滤 | 79%企业使用ATS，但<15%求职者了解筛选逻辑 | ★★★★★ |
| 不知道简历差在哪里 | 用户无法量化自己与岗位的匹配差距 | ★★★★★ |
| 简历同质化严重 | 简历内容雷同，无法体现个人差异化价值 | ★★★★★ |
| 投递决策困难 | 不确定自己是否适合投递某个岗位 | ★★★★☆ |
| 经历表达能力弱 | 有好经历但不知如何用专业语言量化呈现 | ★★★★☆ |
| 简历制作效率低 | 针对每个岗位手动修改简历耗时耗力 | ★★★★☆ |

### 1.3 产品机会

现有市场的简历工具普遍存在以下缺陷：

- **信息收集体验差**：全部采用冷冰冰的表单填写，认知负担高
- **匹配分析浅**：只做关键词匹配，无法做到深度能力维度分析
- **全链路割裂**：JD匹配工具与简历生成工具是两个独立产品，用户需要多工具切换
- **投递决策缺失**：没有工具明确告诉用户"适不适合投"

**JobFit的机会**：打造"诊断→决策→生成"全链路闭环平台，以对话式体验收集信息，深度解析JD，透明输出匹配评分，智能生成匹配简历。

## 2. 用户需求

### 2.1 目标用户画像

#### 主要用户群：社招求职者

| 属性 | 描述 |
|------|------|
| **年龄** | 24-35岁 |
| **身份** | 在职找机会 / 主动跳槽 / 被动离职 |
| **痛点** | 简历投出去没反应，不知道哪里出了问题 |
| **诉求** | 快速生成高质量简历，提升面试邀约率 |
| **行为特征** | 同时投递多个岗位，需要针对每个岗位定制简历 |

#### 次要用户群：应届生

| 属性 | 描述 |
|------|------|
| **年龄** | 22-25岁 |
| **身份** | 应届毕业生 / 实习生 |
| **痛点** | 没有工作经验，不知道如何写简历 |
| **诉求** | 通过AI引导快速建立第一份专业简历 |
| **行为特征** | 对AI工具接受度高，愿意尝试新产品 |

#### 细分用户群：转行者

| 属性 | 描述 |
|------|------|
| **年龄** | 28-40岁 |
| **身份** | 希望跨行业/跨职能转型的职场人 |
| **痛点** | 有经历但不知道如何向新岗位迁移表达 |
| **诉求** | AI帮助重新包装已有经历，发现可迁移技能 |

### 2.2 核心用户需求

| 需求类型 | 具体需求 | 优先级 |
|---------|---------|--------|
| 信息整理 | 快速、轻松地整理个人职业信息 | P0 |
| JD解析 | 自动理解岗位核心要求 | P0 |
| 差距诊断 | 清晰知道自己与岗位的差距在哪里 | P0 |
| 简历生成 | 自动生成匹配岗位的高质量简历 | P0 |
| 投递建议 | 明确知道是否值得投递这个岗位 | P1 |
| 多版本管理 | 针对不同岗位保存多份简历 | P1 |
| 档案沉淀 | 建立个人职业档案，长期使用 | P2 |

### 2.3 典型使用场景

**场景一：社招跳槽**
> 张三，产品经理，工作3年，想换家更大的公司。他在BOSS直聘看到一个心仪岗位，将JD粘贴到JobFit，系统分析他与岗位的匹配度为71分（强烈推荐投递），并指出他缺少"数据分析"相关经验描述。系统帮他生成了一份突出数据分析能力的简历，并在相关工作经历中补充了数据驱动的描述。

**场景二：应届生求职**
> 李四，计算机专业应届生，第一次写简历不知从何下手。通过JobFit的对话式引导，15分钟内完成信息采集，系统根据他投递的算法岗JD生成了一份突出竞赛经历和项目经历的简历，匹配度68分，建议他优化后投递。

**场景三：职场转型**
> 王五，传统行业销售总监，想转型互联网运营。JobFit帮他识别销售经历中的可迁移技能（用户运营、数据分析、KPI管理），重新包装成互联网运营语言，匹配度从初始38分提升到59分。

## 6. 核心功能设计

### 6.1 Module A：信息采集模块

#### 功能目标
以最低认知成本收集用户完整、高质量的职业信息，支持"对话式引导"和"文件导入"两种方式。

#### 设计原则
- **渐进式披露**：分步骤引导，避免信息过载
- **智能追问**：AI根据回答质量判断是否追问
- **实时保存**：对话进度实时保存，支持断点继续
- **可编辑**：所有信息可后续修改

#### 两种信息采集方式

| 方式 | 描述 | 适合人群 |
|------|------|---------|
| 对话式引导（推荐） | AI助手用自然语言逐步引导，用户口语化回答 | 信息尚未整理的新用户 |
| 上传简历文件 | 上传PDF/Word，AI自动解析，确认补充 | 已有简历的用户 |

#### 输入模式：文字 + 语音双模式

对话采集支持**文字输入**和**语音输入**两种模式，用户可随时切换。

| 模式 | 交互方式 | 适用场景 |
|------|---------|---------|
| **文字模式** | 键盘输入，发送消息 | 公共场所、需要精确输入时 |
| **语音模式** | 点击麦克风按钮，说话，AI识别转文字后处理 | 独处时、移动场景、描述复杂经历时 |

**语音模式交互设计：**

```
语音输入状态机：

[待机态]  → 用户点击麦克风 →  [聆听态]
                                  │
                          实时显示音波动画
                          红点闪烁 + "正在聆听"
                                  │
                     用户停顿2秒 或 手动点击停止
                                  │
                             [识别态]
                          AI转写文字（≤2秒）
                                  │
                             [完成态]
                     文字填入输入框，用户可确认或修改
                          → 发送进入对话流

失败处理：
- 识别失败（无声/噪音过大）→ 提示重试，自动回退文字模式
- 网络异常 → 本地缓存音频，恢复后重试
```

**技术实现：**
- 微信小程序端：使用微信 `RecorderManager` 录音 API + 微信语音识别
- Web端：Web Speech API
- 识别结果经大模型语义理解后提取结构化信息（同文字模式）

#### 对话式采集设计（MVP精简版：15问 × 5模块）

> **MVP策略**：聚焦核心信息，降低用户流失。15问在5-8分钟内完成，V1.5迭代至完整25问版本。

| 模块 | 题目数 | 核心问题 | MVP必要性 |
|------|--------|---------|----------|
| 基本信息 | 3题 | 姓名、城市、求职状态 | P0 |
| 教育背景 | 2题 | 最高学历+学校专业、毕业时间 | P0 |
| 工作经历 | 4题×循环 | 公司职位+时间、核心职责、最佳成果（含数字）、使用技术 | P0 |
| 技能 | 2题 | 技术技能+熟练度、语言能力 | P0 |
| 求职意向 | 2题 | 目标岗位类型、期望城市 | P0 |
| 自我介绍 | 1题 | AI基于以上信息自动生成草稿，用户确认/修改 | P1 |

> **完整25问版（V1.5）**：增加薪资期望、项目经历、证书、社会活动、工作模式偏好等维度

> **预估完成时间：MVP版 5-8分钟 | 完整版 10-15分钟**

#### 新手引导设计（Onboarding · 单页合并版）

> **设计决策**：将原3步引导页 + 独立隐私页合并为**单一页面**，减少用户跳转步骤，降低流失。

> **功能定位**：用户无需离开JobFit去复制JD，直接在App内搜索目标岗位，系统后台实时抓取招聘数据，一键引入并进行匹配分析。

```
搜索功能模块
│
```

### 6.4 Module D：简历生成模块

#### 功能目标
根据用户信息档案和JD要求，生成ATS友好、突出成果、高度匹配目标岗位的简历内容。

#### 核心生成策略

> ⚠️ **真实性原则（不可逾越的底线）**：JobFit 只优化表达方式，绝不捏造数据、虚构经历、夸大职级。AI生成内容必须基于用户提供的真实信息，所有数字均来源于用户输入，不得自行补全。违反真实性原则不仅损害用户信任，也存在法律风险。

| 策略 | 描述 | 边界 |
|------|------|------|
| JD关键词植入 | 将JD高权重关键词自然融入简历各模块，Critical关键词100%覆盖 | 只替换近义表达，不捏造经历 |
| STAR法则改写 | 将模糊描述转化为"情境+任务+行动+结果"的量化表述 | 数字必须来自用户提供，无数据时提示用户补充而非自动编造 |
| 动词升维 | 弱动词替换（负责→主导，参与→深度协作，做→构建/设计） | 仅提升表达力，不改变事实内容 |
| 相关性排序 | 最匹配岗位的经历排在靠前位置 | 只调整顺序，不删减真实经历 |
| ATS格式优化 | 单栏布局、标准日期格式、标准学历称谓、联系方式置顶 | 格式标准化，内容不变 |

#### 模板推荐策略

| 岗位类型 | 推荐模板风格 |
|---------|------------|
| 互联网技术 | 极简黑白单栏（ATS兼容性最佳） |
| 产品/运营 | 现代双色单栏 |
| 设计/创意 | 有设计感双栏 |
| 金融/咨询 | 传统保守单栏 |
| 市场/销售 | 成果突出型（数据业绩为核心） |
| 管理/高管 | 商务精英型 |

#### 简历结构排序策略

| 用户类型 | 推荐结构 |
|---------|---------|
| 应届生 | 教育背景 → 实习经历 → 项目经历 → 技能 → 证书 |
| 社招（同行） | 个人简介 → 工作经历 → 技能 → 教育背景 → 证书 |
| 转行 | 个人简介（转行优势）→ 技能 → 相关项目 → 工作经历 → 教育背景 |
| 高管 | 核心成就 → 工作经历 → 教育背景 → 证书 |

#### 国内ATS适配规则

| ATS特性 | 应对策略 |
|---------|---------|
| 关键词精确匹配 | 使用JD原词，不做同义替换 |
| 格式解析 | 单栏布局，避免表格、文本框、图片 |
| 文件格式 | 优先PDF，次选Word |
| 日期格式 | 使用"2022.07-2024.12"格式 |
| 联系方式 | 姓名/电话/邮件放最顶部 |
| 技能格式 | "Python \| Java \| MySQL"（竖线分隔） |

### 6.5 【核心功能】一档案 × 多JD → 多简历（Multi-Resume Engine）

> ⭐ 这是 JobFit 区别于所有竞品的核心能力：用户**只需建立一次档案**，即可针对任意数量的岗位JD，分别生成高度定制化的专属简历，每份简历独立管理、独立优化、独立导出。

#### 6.5.1 功能定位

```
                    ┌─────────────────────────────┐
                    │      用户个人档案（唯一）       │
                    │  工作经历 × 技能 × 项目 × 教育  │
                    └──────────────┬──────────────┘
                                   │ 作为"原材料库"
              ┌────────────────────┼────────────────────┐
              │                    │                    │
              ▼                    ▼                    ▼
       [JD-A 产品经理]      [JD-B 数据分析师]    [JD-C 运营总监]
              │                    │                    │
              ▼                    ▼                    ▼
       简历A（定制版）        简历B（定制版）        简历C（定制版）
       突出产品设计经历        突出数据分析成果        突出增长运营数据
       植入产品类关键词        植入数据类关键词        植入运营类关键词
              │                    │                    │
              ▼                    ▼                    ▼
         独立存储             独立存储              独立存储
         独立导出             独立导出              独立导出
```

#### 6.5.2 核心设计原则

| 原则 | 描述 |
|------|------|
| **档案是原材料库** | 用户的所有经历完整保存，系统按JD需求智能选取，不删减原始信息 |
| **JD驱动内容选择** | 相同的经历，根据不同JD的侧重点，决定"写哪些"和"怎么写" |
| **关键词动态植入** | 每份简历的关键词池来自各自的JD，彼此独立不干扰 |
| **版本完全隔离** | 每份简历独立存储、独立编辑，修改A不影响B和C |
| **生成不替代原档案** | 生成的简历是"输出物"，不会反向修改用户的个人档案 |

#### 6.5.3 内容选取策略（AI决策逻辑）

**第一步：JD能力需求提权**

系统从JD中提取核心能力维度，并赋予优先级权重：

```
JD（产品经理）解析结果：
  Critical：产品规划(0.9)、用户研究(0.85)、数据驱动(0.8)
  Important：项目管理(0.7)、跨团队协作(0.65)
  Optional：技术背景(0.4)
```

**第二步：档案经历匹配打分**

对用户档案中每一条工作经历、项目经历进行相关性评分：

```
经历A（产品设计，3年）    → 相关性 0.92 → ✅ 优先展示
经历B（数据分析项目）     → 相关性 0.75 → ✅ 展示（作为加分项）
经历C（行政管理经历）     → 相关性 0.12 → ❌ 不展示（与JD无关）
```

**第三步：内容角度重写**

即使是同一段经历，针对不同JD采用不同的描述角度：

| 原始经历 | 针对产品经理JD的写法 | 针对运营JD的写法 |
|---------|-------------------|----------------|
| 主导了APP改版项目 | 主导APP全链路产品设计，输出PRD，协调3个研发团队按时交付 | 主导APP改版运营推广，上线首周DAU增长42%，用户留存率提升18% |
| 做了用户调研 | 设计并执行200人用户访谈，提炼核心痛点，驱动产品迭代3个版本 | 深度用户调研200人，洞察核心需求，支撑运营策略优化，转化率提升25% |

**第四步：结构动态排序**

根据JD类型，自动调整简历模块顺序和经历排列：

```
产品经理JD → 个人简介 → 产品经历（最相关优先）→ 技能 → 项目 → 教育
数据分析JD → 个人简介 → 技能（数据技术栈前置）→ 数据项目 → 工作经历 → 教育
运营JD    → 个人简介 → 核心数据成果 → 运营经历 → 技能 → 教育
```

### 6.6 Module E：投递管理模块（新增）

> **功能定位**：简历生成完成后，自动同步至投递管理看板，用户在一个地方追踪所有岗位的求职进展，形成从「生成简历」到「拿到offer」的完整闭环。

#### 6.6.1 自动同步机制

```
简历生成完成
      │
      ▼
自动创建投递记录（status: pending）
      │
      ▼
同步至投递管理看板
      │
      ▼
用户手动更新状态（已投递/面试中/已录用...）
```

**触发时机：** 用户点击「生成简历」并完成生成后，系统自动在投递管理中创建一条记录，与该简历和JD关联，初始状态为「待投递」。

#### 6.6.2 投递状态流转

```
[待投递] → [已投递] → [简历筛选通过] → [面试中] → [已录用 ✓]
                              │                  └──→ [谈薪中]
                              │
                          [未通过]                [已拒绝 ✗]
                              │
                          [已放弃]
```

状态由用户手动更新，系统支持一键快捷切换。

#### 6.6.3 投递管理看板功能

| 功能区域 | 功能点 | 说明 |
|---------|--------|------|
| **全局看板** | 分Tab展示（全部/待投递/面试中/已录用等） | 快速定位不同阶段的投递 |
| **统计概览** | 进行中数量、面试数、录用数、待行动数 | 顶部数据卡片 |
| **投递卡片** | 公司、岗位、匹配分、当前状态、下一步行动 | 列表每行核心信息 |
| **详情页** | 时间轴进展、待行动提醒、关联简历、备注 | 点击卡片进入 |
| **待行动提醒** | ⚡标记需要跟进的投递（如：面试临近、offer截止） | 黄色高亮提示 |
| **关联简历** | 从详情页直接查看/编辑该岗位的专属简历 | 与Module D双向联动 |
| **备注** | 记录面试反馈、薪资谈判、注意事项 | 纯文本，不影响简历 |
| **状态更新** | 快捷按钮一键切换投递状态 | 6种状态快捷选择 |

#### 6.6.4 时间轴设计

## 9. 技术实现建议

### 9.1 技术栈选型

> MVP阶段以**微信小程序为主端**，架构设计保持多端可扩展性，后续无需重构即可接入其他端。

| 层次 | 技术选型 | 说明 |
|------|---------|------|
| **小程序端（主端）** | 微信原生小程序 / Taro | Taro支持未来一套代码多端编译，MVP可用原生快速启动 |
| **Web端（辅助端）** | React 18 / Next.js 14 | SSR支持SEO，与小程序共用后端API |
| **后端框架** | Python FastAPI | 高性能，与AI生态天然兼容；REST API供多端复用 |
| **关系数据库** | MySQL 8.0 | 用户/简历等结构化数据 |
| **缓存** | Redis 7 | 会话状态、AI生成任务队列、高频查询缓存 |
| **文件存储** | 阿里云OSS | 简历文件存储；小程序端通过临时链接访问 |
| **全文搜索** | Elasticsearch 8 | JD语料检索、关键词分析 |
| **对话记录** | MongoDB | 灵活存储对话上下文 |
| **消息队列** | RocketMQ | 异步处理AI生成等耗时任务，小程序轮询结果 |
| **容器化** | Docker + Kubernetes | 弹性伸缩，支撑流量波峰 |
| **文档解析** | Apache Tika + PyMuPDF | PDF/Word文件结构化解析 |
| **CI/CD** | GitHub Actions | 自动化测试和部署 |

#### 多端扩展性设计原则

```
后端 API（统一，无端差异）
    │
    ├── 微信小程序（V1.0 主端）
    ├── Web端       （V1.0 辅助端）
    ├── 浏览器插件  （V2.0 扩展）
    └── iOS/Android （V3.0 扩展）

核心原则：
1. 后端 API 全部 REST + JSON，无端差异
2. 小程序与 Web 共用同一账户体系（微信 OpenID + 手机号绑定）
3. 数据完全互通，任意端操作实时同步
4. 新增端只需接入现有 API，后端零改造
```

### 9.2 大模型驱动架构

> JobFit 以大模型为核心驱动力，贯穿产品全链路。不同模块根据任务特性选用最合适的调用方式。

#### 大模型调用链路

```
用户输入
   │
   ├──[对话采集]──→ LLM（对话管理）
   │                  ├── System Prompt：你是职业信息采集助手，任务是通过自然对话提取用户职业信息...
   │                  ├── Function Calling：每轮对话后调用 extract_profile_fields() 结构化输出
   │                  └── 输出：标准化UserProfile JSON
   │
   ├──[JD解析]───→ LLM（信息提取）
   │                  ├── System Prompt：你是JD分析专家，将招聘文本解析为结构化能力需求...
   │                  ├── Function Calling：调用 parse_jd_fields() 输出标准JD数据模型
   │                  └── 输出：标准化JobDescription JSON
   │
   ├──[匹配评分]──→ LLM（语义理解）+ 规则引擎
   │                  ├── LLM负责：软技能语义匹配、隐性要求识别、经历相关性判断
   │                  ├── 规则引擎负责：技能精确匹配、年限计算、学历对比
   │                  └── 输出：匹配评分 + 缺口分析 + 投递建议
   │
   └──[简历生成]──→ LLM（内容创作）
                      ├── System Prompt：你是资深简历优化专家，基于用户真实经历和目标JD生成简历...
                      ├── 约束条件：不捏造数据、保持真实性、STAR法则、关键词覆盖
                      ├── 输入：UserProfile + JD解析结果 + 匹配缺口报告
                      └── 输出：完整简历内容（各模块文字）
```

|---------|---------|---------------|------|

| JD语义解析 | GPT-4o / Claude 3.5 | DeepSeek V3 | 信息提取精度要求高 |

#### 模型降级策略

```

- JD解析：TF-IDF关键词提取（无语义理解）

```

### 9.3 关键技术方案

#### 对话管理
- 基于多轮对话上下文管理（sliding window策略，保留最近10轮）
- 每轮对话结束后调用 Function Calling 进行实体抽取（NER），结构化存储到档案
- 对话上下文存储在 MongoDB，支持断点恢复
- 微信小程序端同时做本地 Storage 缓存，防止意外退出丢失进度

#### JD解析流程
```
JD原文 → 文本清洗（去广告/格式化字符）→ 分句
       → LLM Function Calling 提取结构化字段
       → 规则校验（年限/学历格式标准化）
       → 关键词权重计算（词频 × 位置权重 × 语义强度）
       → 输出标准化 JobDescription JSON
```

#### 匹配算法
```
[精确匹配层]
用户技能列表 ∩ JD技能要求 → 技能覆盖率（规则计算）

[语义匹配层]
用户经历描述向量化 + JD职责要求向量化
→ 余弦相似度计算（语义相关性）

[LLM判断层]
软技能匹配、隐性要求识别、经历迁移性评估
→ GPT-4o-mini 按固定 Prompt 输出结构化分数

[加权融合]
三层结果 × 岗位类型权重 → 最终评分（0-100）
```

#### 简历生成流程
```
输入：UserProfile + JD解析结果 + 匹配缺口报告
    ↓
Prompt构建：
  System：资深简历专家角色 + 真实性约束 + 格式要求
  User：用户档案信息 + JD要求 + 需强调的维度
    ↓
GPT-4o 生成草稿
    ↓
内容验证：
  ① 真实性检测（生成内容与原始档案对比，偏差>30%告警）
  ② 关键词密度检查（Critical词100%覆盖，单词≤4次）
  ③ 格式合规检查（ATS规则验证）
    ↓
输出：结构化简历内容 JSON + 优化操作日志
```

### 9.5 数据隐私与安全（重点章节）

> 用户的简历信息是高度敏感的个人数据，包含真实姓名、联系方式、工作经历、收入信息等。JobFit 将数据安全作为产品的核心设计原则之一，而非事后合规。

#### 隐私设计原则

| 原则 | 具体措施 |
|------|---------|
| **数据最小化** | 只收集生成简历必需的信息，不收集与功能无关的数据（如通讯录、位置等） |
| **明示授权** | 微信小程序首次使用前显示隐私政策弹窗，用户主动同意后才开始信息收集 |
| **用户控制权** | 用户可随时查看、修改、删除自己的所有数据，包括档案、JD记录、简历 |
| **不用于训练** | 用户数据不用于AI模型训练，不向第三方共享，不用于广告 |
| **最短保留** | 用户注销账户后，数据在30天内完全删除（含备份） |

#### 微信小程序隐私合规

```
必须实现（微信平台强制要求）：
① 隐私政策链接（小程序设置中配置）
② 首次使用时用户服务协议+隐私政策弹窗（用户点击同意后才能继续）
③ 涉及敏感权限时的单独授权：
   - 手机号：使用微信官方手机号授权组件
   - 相机/相册（上传简历）：使用时才申请，不提前申请
④ 个人信息收集清单（在隐私政策中列明所有收集项）

敏感信息收集范围声明：
  收集：姓名、手机号、邮箱、工作经历、教育背景、技能信息
  不收集：通讯录、位置、设备标识符、生物特征
  使用目的：生成求职简历，不用于其他用途
```

#### 数据存储安全

| 安全措施 | 实现方式 |
|---------|---------|
| 传输加密 | HTTPS/TLS 1.3全链路加密 |
| 存储加密 | 敏感字段（手机号/邮箱）AES-256加密存储 |
| 访问控制 | 严格的用户数据隔离，API层用户身份校验，防止越权访问 |
| AI调用隔离 | 调用大模型API时，仅传递当次必要信息，不携带用户ID |
| 简历文件 | OSS存储，私有Bucket，仅通过有时效的签名URL访问 |
| 日志脱敏 | 系统日志中手机号、邮箱等字段自动脱敏（138****1234） |
